# 智能考研系统可行性分析报告

## 1. 项目概述

本项目为“智能考研系统”，现有代码实现为前后端分离架构：后端使用 Spring Boot 3.3.5，前端使用 Vue 3 + Vite，默认以 H2 文件数据库和本地上传目录保存数据与文件。系统面向个人考研学习场景，围绕“资料管理、知识库问答、错题管理、学习计划、AI 设置”提供功能。

系统已实现的核心能力包括：用户注册登录与 JWT 鉴权；资料文件夹三级管理；PDF、Word、图片、文本等资料上传与文本抽取；知识片段切分；基于 Elasticsearch 与 embedding 的混合检索增强问答；无模型 API Key 时的本地兜底摘要；错题录入、附件管理、掌握状态与随机练习；手动学习计划与 AI 规划草稿；用户级 AI 模型配置。

## 2. 技术可行性

### 2.1 已具备的技术基础

项目技术栈成熟，当前实现与毕业设计规模相匹配：

| 层次 | 实际技术 | 可行性说明 |
| --- | --- | --- |
| 前端 | Vue 3.5、Vite 5、lucide-vue-next | 适合构建单页学习工作台，已实现登录、资料、问答、计划、错题、设置页面 |
| 后端 | Spring Boot 3.3、Spring Security、Spring Data JPA | 适合实现 REST API、鉴权、业务服务与数据访问 |
| 数据库 | H2 文件数据库 | 便于本地演示和毕业设计部署，后续可迁移 MySQL/PostgreSQL |
| 文件处理 | PDFBox、Apache POI、Tesseract OCR | 覆盖 PDF、Word、图片、文本抽取的主要需求 |
| AI 接入 | OpenAI 兼容 Chat Completions / Embeddings 接口 | 支持用户自填模型、Endpoint、API Key，可兼容 OpenAI、DeepSeek、通义等兼容接口 |
| 检索 | Elasticsearch 可选启用 | 支持关键词检索、向量检索、RRF 融合；不可用时回退数据库本地检索 |

### 2.2 关键技术实现可行性

1. 用户认证可行：后端已有 `AuthController`、`AuthService`、`JwtService`、`JwtAuthenticationFilter`，使用 JWT 保护除注册登录外的 API。
2. 资料管理可行：`StudyFolder` 支持父子文件夹和深度字段，`StudyFile` 保存文件元信息、抽取文本、是否加入知识库等状态。
3. 文本抽取可行：`TextExtractionService` 已按文件类型调用 PDFBox、Apache POI、Tesseract 或直接读取文本。
4. 知识库构建可行：`FileService.rebuildKnowledge()` 将文本按约 800 字符切片，120 字符重叠，保存为 `KnowledgeChunk`。
5. 智能问答可行：`ChatService` 以当前文件夹及子文件夹为范围，最多选取 5 个片段构造 prompt，支持普通 HTTP 和 SSE 流式输出。
6. 检索增强可行：`ElasticsearchService` 支持关键词检索、向量检索、RRF 融合；若 ES 或 embedding 不可用，系统仍可用数据库片段进行本地关键词评分。
7. 错题管理可行：`MistakeService` 支持错题文本、题目/解析文件、题目/解析图片附件、掌握状态、自定义状态、科目标签和随机练习。
8. 学习计划可行：`StudyPlanService` 支持周范围查询、增删改；`StudyPlanAiService` 支持 AI 对话、生成操作草稿、确认后落库。

### 2.3 技术风险与边界

| 风险 | 当前状态 | 应对方式 |
| --- | --- | --- |
| OCR 质量依赖本地 Tesseract | 代码调用本地 `tesseract` 命令 | 手动校正文档文本；部署时安装中文语言包 |
| H2 不适合多人长期生产使用 | 当前默认 H2 文件库 | 毕设演示可用；生产化迁移到 MySQL/PostgreSQL |
| AI 结果受第三方模型影响 | 用户自填 API Key 和 Endpoint | 提供本地检索兜底；文档中明确外部模型不是系统内置能力 |
| Elasticsearch 为可选服务 | 配置默认启用但本机需另行启动 | 不可用时回退数据库检索，保证核心问答流程可演示 |
| 文件存储为本地目录 | 默认 `./uploads` | 适合单机部署；后续可扩展对象存储 |

结论：从现有代码和依赖看，系统核心功能已具备实现基础，技术路径可行。

## 3. 经济可行性

### 3.1 开发成本

本系统使用开源框架和本地运行环境，开发成本较低：

- Spring Boot、Vue、Vite、H2、PDFBox、Apache POI 均可免费使用。
- 本地演示无需采购服务器，可在普通 Windows 电脑上运行。
- AI 能力采用用户自备 API Key，不要求项目方承担模型调用费用。
- Elasticsearch 是可选增强项，毕业设计演示可根据机器条件决定是否启动。

### 3.2 运行成本

| 成本项 | 说明 |
| --- | --- |
| 服务器/本机资源 | 后端、前端、H2、上传目录可单机运行 |
| OCR | 本地 Tesseract，无按量费用 |
| 大模型调用 | 用户自填 API Key，费用取决于所选模型服务商 |
| 向量检索 | 有 embedding API Key 时会产生外部调用费用；无 Key 时退化为关键词检索 |
| 存储 | 本地文件和 H2 数据文件占用磁盘空间 |

### 3.3 经济价值

系统面向个人考研备考，价值主要体现在：

- 降低资料整理成本：将教材、笔记、习题、资料统一归档到文件夹。
- 提高复习效率：通过知识问答快速定位资料片段。
- 加强错题复盘：错题按掌握状态、标签和随机练习组织。
- 辅助计划管理：以周课表方式安排自习、复盘、考试、任务和休息。
- 支持低成本演示：无外部 AI Key 时仍可展示资料管理、错题、计划和本地检索摘要。

结论：系统采用低成本开源技术栈，经济投入与毕业设计项目规模匹配，具备经济可行性。

## 4. 社会因素可行性

### 4.1 用户接受度

考研学习中常见痛点包括资料分散、笔记难检索、错题复盘不系统、计划执行不稳定。本系统围绕这些痛点设计，功能与目标用户的日常学习行为一致，用户接受门槛较低。

### 4.2 法律与伦理因素

- 系统要求用户上传个人学习资料，应提示用户只上传自己有权使用的资料。
- API Key 由用户自行配置，系统不应公开展示或泄露密钥。
- AI 回答可能存在不准确，系统通过“来源引用”和“资料不足时说明”降低误导风险。
- 本系统定位为学习辅助工具，不替代教师指导或正式考试评价。

### 4.3 数据安全因素

当前已实现用户隔离：多数数据表通过 `owner` 或文件夹归属关联到当前用户，服务层校验资源所有权。登录态使用 JWT。需注意的是，默认 JWT Secret 在配置中是演示值，生产部署时必须更换。

结论：在明确资料版权、隐私保护和 AI 辅助边界的前提下，本系统具备社会可行性。

## 5. 可选实施方案比较

| 方案 | 描述 | 优点 | 缺点 | 适用性 |
| --- | --- | --- | --- | --- |
| 方案 A：传统资料管理系统 | 只做文件上传、分类、搜索 | 实现简单，成本最低 | 缺少智能问答和计划辅助，创新性不足 | 不推荐作为最终方案 |
| 方案 B：完全依赖大模型问答 | 用户资料直接交给模型回答 | 交互自然，开发较快 | 成本高，来源不可追溯，长文档受上下文限制 | 不适合作为学习资料系统主方案 |
| 方案 C：轻量 RAG + 本地兜底 | 文件抽取切片，检索后再交给模型；无模型时本地摘要 | 来源可追踪，成本可控，系统已有实现 | 需要处理文本抽取、检索质量和模型配置 | 推荐 |
| 方案 D：企业级知识库平台 | MySQL/PostgreSQL、对象存储、队列、权限管理、部署集群 | 扩展性强 | 超出毕业设计规模，部署复杂 | 可作为后续升级方向 |

## 6. 选定方案与理由

本项目选择“方案 C：轻量 RAG + 本地兜底”的实施方案。

选择理由：

1. 与已实现代码一致：项目已具备文件抽取、知识片段、混合检索、AI 问答、来源引用和本地兜底能力。
2. 技术复杂度适中：既能体现 Spring Boot、Vue、数据库、文件处理、AI 接入和检索增强，又不会过度依赖复杂分布式系统。
3. 成本可控：没有 API Key 也能演示核心流程；有 API Key 时提升回答质量。
4. 学习价值明确：资料管理、错题复盘和计划管理均服务于考研学习闭环。
5. 可扩展性较好：后续可替换数据库、接入对象存储、增加权限角色、升级检索服务。

## 7. 可行性结论

综合技术、经济和社会因素，本项目具备实施可行性。系统当前实现已覆盖毕业设计中要求展示的核心模块，且有清晰的扩展方向。建议后续重点完善部署说明、测试用例、数据备份、API Key 加密存储和生产数据库迁移方案。
