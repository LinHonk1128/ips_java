# 智能考研系统可行性分析报告

## 1. 项目概述

智能考研系统是一个面向个人考研备考场景的 Web 应用。系统采用前后端分离架构，前端基于 Vue 3 和 Vite 构建学习工作台，后端基于 Spring Boot 3 提供 REST API、JWT 鉴权、文件处理、知识库问答、学习画像、学习计划和错题复盘等能力。数据主要保存于 MySQL，本地上传资料保存于 `uploads` 目录，Elasticsearch、Tesseract OCR 和 OpenAI 兼容模型接口作为增强能力按需接入。

系统当前围绕“资料进入知识库、问答引用知识片段、反馈形成学习画像、薄弱点进入计划和错题复习”形成学习闭环。用户可以注册登录，完成考研日期和学科初始化，上传 PDF、Word、图片或文本资料，校正文档抽取文本，将资料切分为知识片段；随后可围绕选定文件夹进行知识问答、定制练题、来源反馈、会话生成笔记；还可维护错题、随机练习并把练习结果回写到关联知识片段；学习规划模块支持手动周计划和 AI 规划草稿。

## 2. 技术可行性

### 2.1 技术栈成熟度

| 层次 | 当前技术 | 可行性说明 |
| --- | --- | --- |
| 前端 | Vue 3.5、Vite 5、ECharts、lucide-vue-next | 适合构建单页学习工作台，已覆盖首页、知识库、知识画像、计划、错题、个人设置和 AI 设置 |
| 后端 | Spring Boot 3.3、Spring MVC、Spring Security、Spring Data JPA | 适合实现分层 REST 服务、无状态鉴权、数据持久化和业务编排 |
| 数据库 | MySQL 8.x | 能稳定保存用户、文件夹、资料、知识片段、错题、计划和 AI 设置 |
| 文件抽取 | PDFBox、Apache POI、Tesseract OCR | 覆盖 PDF、Word、图片和纯文本资料的抽取场景 |
| 检索增强 | Elasticsearch 可选启用 | 支持关键词检索、向量检索和 RRF 融合；不可用时回退数据库检索 |
| AI 接入 | OpenAI 兼容 Chat Completions / Embeddings 接口 | 用户级模型、Endpoint、API Key 可配置，便于兼容不同服务商 |
| 部署形态 | 单机开发/演示环境 | 后端、前端、MySQL、上传目录可在普通个人电脑上运行 |

### 2.2 核心功能实现可行性

1. 认证与数据隔离可行。后端通过 `AuthController`、`AuthService`、`JwtService` 和 `JwtAuthenticationFilter` 完成注册、登录、JWT 解析和用户身份注入。业务接口除注册登录外均要求登录，服务层按当前用户校验资源归属。
2. 资料管理可行。`StudyFolder` 支持父子层级和最大 3 级深度，`StudyFile` 保存文件标签、原始文件名、存储路径、内容类型、抽取文本和知识库启用状态。
3. 文本抽取可行。`TextExtractionService` 对 PDF、DOC/DOCX、图片、TXT/Markdown 分别使用 PDFBox、Apache POI、Tesseract 或直接读取方式处理，并提供失败提示。
4. 知识库构建可行。`FileService` 将资料抽取文本按标题、段落、句子和自然停顿进行语义切片，当前切片版本为 2，目标长度约 800 字符，最大约 1100 字符。
5. 知识问答可行。`ChatService` 在当前文件夹及子文件夹范围内检索候选知识片段，经过 rerank 和多样性选择后最多送入 5 个片段，支持普通问答、SSE 流式问答、深度回答和来源引用。
6. 学习画像可行。`KnowledgeChunk` 记录引用次数、正确反馈、错误反馈、最近访问和最近练习时间，`KnowledgeProfileService` 聚合总览、学科、资料、趋势、分布、活动、风险和诊断建议。
7. 错题闭环可行。`MistakeService` 支持题干/解析文本、题目文件、解析文件、图片附件、自定义状态、科目标签、随机练习、定制题加入错题和练习结果回写知识片段。
8. 学习计划可行。`StudyPlanService` 支持周视图 CRUD，`StudyPlanAiService` 通过 AI 生成 CREATE/UPDATE/DELETE 草稿，用户确认后才写入真实日程。

### 2.3 技术风险与应对

| 风险 | 当前状态 | 应对方式 |
| --- | --- | --- |
| OCR 质量不稳定 | 图片 OCR 依赖本机 Tesseract 和语言包 | 支持人工校正抽取文本，部署前确认 `tesseract` 命令可用 |
| 外部模型不可用或成本波动 | AI 功能依赖用户配置的兼容接口 | 知识库问答提供本地摘要兜底，直接聊天和 AI 计划明确提示配置模型 |
| Elasticsearch 未启动 | ES 是增强检索服务 | 代码在 ES 不可用时短期跳过并回退数据库关键词检索 |
| API Key 安全 | 当前面向本地学习/演示，支持用户级保存 | 正式部署应使用环境变量或密钥管理，并对数据库中的 Key 加密 |
| 单机文件存储扩展性有限 | 上传资料保存在本地目录 | 毕设演示足够，后续可扩展对象存储和备份策略 |
| 数据库配置依赖本机环境 | 默认连接本机 MySQL | 文档和配置中保留可修改入口，演示前确认账号、密码和库名 |

综上，项目采用成熟框架和可降级的增强能力，核心技术路线可行。

## 3. 经济可行性

### 3.1 开发成本

系统使用开源框架和常见开发工具：Spring Boot、Vue、Vite、MySQL Community Server、PDFBox、Apache POI、ECharts 等均可免费使用。开发和演示不要求采购专用服务器，普通 Windows、macOS 或 Linux 电脑即可运行。外部 AI 调用采用用户自备 Key 的方式，不把模型费用固化为系统运行前提。

### 3.2 运行成本

| 成本项 | 说明 |
| --- | --- |
| 本机资源 | 后端、前端、MySQL 和上传目录可单机运行 |
| 文件存储 | 主要消耗本地磁盘空间 |
| OCR | 本地 Tesseract，无按量费用 |
| AI 聊天 | 取决于用户选择的模型服务商和调用频率 |
| Embedding | 开启向量检索时可能产生外部调用费用 |
| Elasticsearch | 可选本地服务，消耗额外内存和磁盘 |

### 3.3 经济价值

系统能降低考研资料整理和复习管理成本：资料统一归档，资料问答有来源追溯，错题复习有状态和标签，学习计划可以结合薄弱点生成建议。由于没有强制云服务依赖，即使只在本地运行，也能完成毕业设计演示和个人学习辅助。

结论：系统投入低、依赖可控、功能价值明确，经济上可行。

## 4. 社会与使用可行性

### 4.1 用户接受度

目标用户是个人考研学习者，其常见痛点包括资料分散、长文档难检索、错题复盘不连续、学习计划难坚持。本系统的模块划分与实际学习行为一致：先整理资料，再围绕资料问答和抽问，通过反馈形成画像，再把薄弱点加入错题和计划复习，用户理解成本较低。

### 4.2 法律与伦理因素

- 用户应只上传自己有权使用的教材、笔记、题目和解析资料。
- 系统定位为学习辅助工具，AI 回答和诊断建议不能替代教师指导或正式评价。
- 问答结果需要尽量展示来源片段，资料不足时应明确说明无法确认。
- API Key 属于敏感信息，正式部署应避免明文暴露、日志输出和仓库提交。

### 4.3 数据安全因素

当前系统通过 JWT 登录态和服务层资源归属校验隔离不同用户数据。密码使用 BCrypt 哈希保存。生产环境仍需进一步处理默认 JWT Secret、API Key 加密、HTTPS、数据库权限、备份恢复和日志脱敏等问题。

结论：在明确版权、隐私和 AI 使用边界后，系统具备社会可行性。

## 5. 实施方案比较

| 方案 | 描述 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- | --- |
| 传统资料管理 | 只做文件夹、上传、搜索 | 简单稳定 | 创新性和学习闭环不足 | 不作为最终方案 |
| 纯大模型问答 | 直接把资料或问题交给模型 | 交互自然 | 成本高，来源弱，长文档受上下文限制 | 不适合作为主方案 |
| 轻量 RAG + 学习闭环 | 抽取切片、检索增强问答、来源反馈、画像、错题和计划联动 | 成本可控，来源可追踪，符合当前代码 | 需要处理抽取、切片和检索质量 | 推荐方案 |
| 企业级知识库平台 | 对象存储、队列、集群、权限体系 | 扩展性强 | 超出毕业设计复杂度 | 后续升级方向 |

本项目选择“轻量 RAG + 学习闭环”方案。该方案既能体现完整的软件工程实现，又能保持本地演示可控性。

## 6. 可行性结论

从技术、经济、社会和实施复杂度看，智能考研系统具备完成和演示条件。当前代码已经覆盖资料管理、知识库问答、学习画像、错题复盘、学习规划、AI 设置和演示数据初始化等核心内容，后续重点应放在测试用例补充、配置安全、部署说明、备份机制和模型密钥管理上。
