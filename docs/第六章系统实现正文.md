# 第6章 系统实现

前文已经完成系统需求分析、总体设计和数据库设计。本章按照第4章确定的功能模块和业务流程，结合前端页面、后端接口、业务服务及数据持久化代码，说明各项功能的具体实现。对于语义切片、混合检索、画像计算和学习闭环等设计规则，本章采用交叉引用方式说明，不再重复其原理与推导。

## 6.1 系统实现概述

系统采用前后端分离方式实现。前端基于 Vue 3 和 Vite 构建，负责页面交互、接口调用和结果展示；后端基于 Spring Boot 3.3.5 构建，使用 Spring MVC 提供 RESTful API，使用 Spring Security 和 JWT 完成身份认证，使用 Spring Data JPA 管理业务数据。MySQL 保存用户、学习档案、资料文件、知识片段、学习行为、错题和学习计划等结构化数据，原始资料及错题附件保存在服务器上传目录。Elasticsearch、Embedding 服务、大语言模型和 OCR 服务分别用于知识检索、文本向量化、答案生成和图片文字识别。

后端按 Controller、Service、Repository、Model 和 DTO 分层组织，前端接口统一封装在 `frontend/src/api/client.js`，业务状态集中在 `useSmartExamApp.js`。本章代码段仅保留业务判断、状态更新和跨服务调用。

## 6.2 用户学习档案与基础支撑实现

用户学习档案与基础支撑为各核心业务模块提供身份、数据范围和考研基础信息。该部分不直接完成知识问答或复习诊断，但决定了资料、错题、画像和计划数据的归属，是个人知识库能够安全运行的前提。

### 6.2.1 用户注册登录实现

注册和登录接口分别为 `POST /api/auth/register` 和 `POST /api/auth/login`。`AuthService.register()` 校验用户名后创建 `User`，密码通过 `BCryptPasswordEncoder` 加密保存；`AuthService.login()` 使用 `PasswordEncoder.matches()` 验证密码，并调用 `JwtService.createToken()` 生成有效期为24小时的 JWT。

前端通过 `setSession()` 保存令牌和用户信息，统一请求方法自动携带 JWT，退出登录时由 `clearSession()` 清除会话数据。

### 6.2.2 JWT 鉴权与用户数据隔离实现

系统采用无状态的 JWT 鉴权方式，不在服务器端保存用户会话。注册和登录请求可以直接访问，其他业务请求均需要通过身份认证。前端访问受保护接口时，将登录后获得的 JWT 放入请求头中发送至后端。系统从请求头中提取令牌，并依次校验令牌格式、HS256 签名和过期时间。校验通过后，系统读取令牌中的用户编号和用户名，将其写入当前请求的安全上下文；令牌缺失、签名错误或已经过期时，请求无法获得有效的认证身份。

业务接口处理请求时，根据安全上下文中的用户编号查询当前用户，并以该用户作为后续数据访问的范围条件。资料目录和错题列表按照用户编号进行查询，学习计划按照用户编号和日期范围进行查询；在查看、修改或删除单条数据时，系统同时校验资源编号与所属用户编号。资料文件的归属关系则通过其所在目录对应的用户进行判断。通过在查询和操作两个环节加入用户归属条件，系统能够避免用户通过修改资源编号访问他人的资料、错题或学习计划，实现不同用户之间的数据隔离。

### 6.2.3 学习档案初始化实现

前端通过 `GET /api/study-profile` 获取初始化状态，并通过 `POST /api/study-profile/onboarding` 提交考研日期和备考科目。`StudyProfileService.onboard()` 保存 `UserStudyProfile`，同时创建带有 `subjectFolder` 和 `subjectOrder` 属性的一级学科文件夹。

更新档案时，`StudyProfileService.update()` 同步调整学科文件夹。删除学科仅取消文件夹的学科标记，不删除其中的资料。考研日期和学科信息供画像与计划模块读取。

## 6.3 资料分类与上传管理模块实现

资料分类与上传管理模块对应第3章的资料管理需求和第4章的资料处理设计。该模块由 `FolderService`、`FileService`、`TextExtractionService` 和 `ElasticsearchService` 共同实现，依次完成目录管理、文件保存、文本抽取、知识片段持久化和检索索引更新。

### 6.3.1 资料文件夹与文件上传实现

文件夹接口位于 `/api/folders`。`FolderService` 在创建或修改文件夹时校验用户归属，并维护父子关系和文件夹深度。

资料通过 `POST /api/folders/{folderId}/files` 上传。前端使用 `FormData` 提交文件和资料标签；后端通过 `folderService.requireOwned()` 校验文件夹归属，并按“用户编号/文件夹编号”保存文件。文件名使用 UUID 处理同名冲突。

文件保存后，系统创建 `StudyFile` 并调用文本抽取服务。抽取成功后重建知识片段；抽取失败时删除已保存文件并终止处理。

### 6.3.2 多类型资料文本抽取实现

原始文件和 `StudyFile` 记录保存后，`FileService` 调用文本抽取服务生成可编辑正文。

文本抽取由 `TextExtractionService.extract()` 根据文件扩展名和内容类型选择处理方式。PDF 文件使用 PDFBox 的 `PDFTextStripper` 读取文本；DOCX 文件使用 Apache POI 的 `XWPFDocument` 按文档正文元素顺序处理段落和表格，并保留加粗、斜体、下划线及表格结构的基础 HTML 标记；DOC 文件使用 `HWPFDocument` 和 `WordExtractor` 读取；TXT、Markdown 和 CSV 文件按 UTF-8 编码直接读取并转换为段落；图片文件则调用 Tesseract OCR。

不同文件类型的分派逻辑如代码6-2所示。系统先根据扩展名判断可直接解析的文档类型，再根据 MIME 类型判断图片。无法自动处理的类型不会直接进入知识库，而是返回提示信息，要求用户补充正文。

代码6-2 多类型资料抽取分派代码

```java
public String extract(Path path, String originalName,
                      String contentType) {
    String lower = originalName.toLowerCase(Locale.ROOT);
    try {
        // 根据扩展名分派到对应的文档解析方法
        if (lower.endsWith(".pdf")) {
            return extractPdf(path);
        }
        if (lower.endsWith(".docx")) {
            return extractDocx(path);
        }
        if (lower.endsWith(".doc")) {
            return extractDoc(path);
        }
        if (lower.endsWith(".txt")
                || lower.endsWith(".md")
                || lower.endsWith(".csv")) {
            return plainTextToHtml(
                    Files.readString(path, StandardCharsets.UTF_8));
        }
        if (contentType != null
                && contentType.startsWith("image/")) {
            return extractImageWithOcr(path);
        }
        // 未支持的类型交由用户补充正文
        return "当前文件类型暂不支持自动抽取。"
                + "请在这里粘贴或编辑正文，然后保存为知识库。";
    } catch (Exception ex) {
        return "自动抽取失败：" + ex.getMessage()
                + "\n请手动编辑正文后，再保存为知识库。";
    }
}
```

图片识别时，系统通过 `ProcessBuilder` 调用 Tesseract 命令，并指定 `chi_sim+eng` 语言模型，以同时支持中文和英文内容。Tesseract 路径可以通过系统配置或 `TESSERACT_CMD` 环境变量指定，程序还会检查 Windows 常见安装目录。如果 OCR 命令不可用或中文语言包未安装，服务返回明确错误信息，前端提示用户处理识别环境。对于错题模块，系统还提供独立识别接口，识别过程使用临时文件，完成后立即删除，不会把临时识别文件保留在用户资料目录。

前端在资料编辑页面显示抽取结果，用户可以修正 OCR 错字、删除页眉页脚或补充缺失内容。保存操作调用 `PUT /api/files/{fileId}`，`FileService.updateText()` 更新资料名称、标签和正文，并重新构建知识片段。人工校正流程与第4章资料处理设计一致。

### 6.3.3 语义知识切片实现

文本抽取结果经过用户校正后，`FileService.rebuildKnowledge()` 先删除该文件的旧知识片段，再将 HTML 内容转换为可检索文本。如果资料未启用知识库或正文为空，则同步删除 Elasticsearch 中对应索引。对于有效正文，系统调用 `splitIntoSemanticChunks()` 生成知识片段。

语义切片的边界规则和参数已经在第4章资料处理设计中说明。本节重点说明代码执行过程：`splitIntoTextUnits()` 先把正文转换为标题、段落和句子单元，`splitIntoSemanticChunks()` 依次合并文本单元，`addChunk()` 生成片段对象，`mergeShortTail()` 处理末尾过短片段。生成结果由 `rebuildKnowledge()` 写入 `KnowledgeChunk` 表，并保存资料编号、文件夹编号、片段序号、页码、切片版本和正文。

语义切片的主循环如代码6-3所示。代码中的边界判断由辅助方法完成，主循环只负责片段生成、重叠内容保留和结果收集。

代码6-3 语义知识切片关键代码

```java
private List<SemanticChunk> splitIntoSemanticChunks(String text) {
    // 将正文转换为可组合的标题、段落和句子单元
    List<TextUnit> units = splitIntoTextUnits(text);
    if (units.isEmpty()) {
        return List.of();
    }

    List<SemanticChunk> chunks = new ArrayList<>();
    List<TextUnit> current = new ArrayList<>();
    boolean currentFromOverlap = false;

    for (int i = 0; i < units.size(); i++) {
        TextUnit unit = units.get(i);
        boolean hasMore = i < units.size() - 1;

        // 根据第4章确定的切片边界决定是否保存当前片段
        if (shouldStartNewChunk(current, unit)) {
            if (!currentFromOverlap) {
                addChunk(chunks, current);
            }
            current.clear();
            currentFromOverlap = false;
        } else if (!current.isEmpty()
                && contentLength(current, unit) > MAX_CHUNK_SIZE) {
            addChunk(chunks, current);
            current = overlapTail(current);
            currentFromOverlap = !current.isEmpty();
            if (contentLength(current, unit)
                    > MAX_CHUNK_SIZE) {
                current.clear();
                currentFromOverlap = false;
            }
        }

        current.add(unit);
        currentFromOverlap = false;
        if (contentLength(current) >= TARGET_CHUNK_SIZE
                && isNaturalStop(unit)) {
            addChunk(chunks, current);
            current = hasMore
                    ? overlapTail(current)
                    : new ArrayList<>();
            currentFromOverlap =
                    hasMore && !current.isEmpty();
        }
    }

    if (!current.isEmpty()) {
        addChunk(chunks, current);
    }
    // 合并末尾过短片段，避免产生无效知识单元
    return mergeShortTail(chunks);
}
```

### 6.3.4 向量生成与检索索引实现

知识片段持久化完成后，`FileService` 调用 `ElasticsearchService.reindexFile()` 重建文件索引。索引文档保存片段编号、文件编号、文件夹编号、用户编号、文件名、片段序号、正文、上传时间和向量数据。用户编号和文件夹编号用于限定检索范围。

`EmbeddingService` 负责生成片段向量。若向量生成失败，索引仍保存文本字段，关键词检索可以继续使用。MySQL 中的 `KnowledgeChunk` 保留完整业务数据，索引重建失败时不会删除已经持久化的知识片段。

## 6.4 个人资料库的智能问答模块实现

个人资料库智能问答模块先在用户选定的资料范围内检索知识片段，再将筛选结果作为模型上下文。主要实现类为 `ChatController`、`ChatService`、`ElasticsearchService`、`EmbeddingService` 和 `KnowledgeChunkInteractionService`。

### 6.4.1 问答交互与请求处理实现

前端问答页面由 `ChatView.vue` 展示，主要状态和操作由 `useSmartExamApp.js` 管理。用户选择知识库问答时，需要先指定资料文件夹，前端将问题、文件夹编号和是否显示来源引用等参数提交给 `ChatController`。控制器获取当前登录用户后调用 `ChatService`，避免由前端直接传入用户编号。

`ChatService` 根据文件夹编号调用 `folderService.requireOwned()` 校验资料范围，再执行知识片段检索、上下文构造和答案生成。返回对象同时包含回答文本与来源列表，前端据此分别渲染回答区域和来源区域。

### 6.4.2 混合检索与候选筛选实现

`ChatService.retrieve()` 首先获取当前文件夹及其全部子文件夹编号，以该范围限制检索。随后调用 `ElasticsearchService.hybridSearch()` 并行执行关键词检索和向量检索。关键词检索使用 `multi_match`，正文 `content` 字段设置更高权重，同时检索文件名；向量检索先将用户问题转换为查询向量，再对 `dense_vector` 字段执行 KNN 检索。两路检索均先返回20个候选片段。

RRF 的计算原理已经在第4章系统设计中说明，本节只说明其代码落地。`hybridSearch()` 使用两个异步任务并行获取关键词结果和向量结果，待两路任务完成后调用 `reciprocalRankFusion()` 合并片段编号。关键代码如代码6-4所示。

代码6-4 混合检索与 RRF 融合关键代码

```java
public List<Long> hybridSearch(Long userId,
                               List<Long> folderIds,
                               String question,
                               AiSettingsResponse settings) {
    // 并行执行关键词召回和向量召回
    var keywordFuture = CompletableFuture.supplyAsync(
            () -> keywordSearch(userId, folderIds, question));
    var vectorFuture = CompletableFuture.supplyAsync(() -> {
        List<Float> vector =
                embeddingService.embed(question, settings);
        return vector.isEmpty()
                ? List.of()
                : vectorSearch(userId, folderIds, vector);
    });
    return reciprocalRankFusion(keywordFuture.join(),
            vectorFuture.join());
}

private List<Long> reciprocalRankFusion(
        List<Long> keywordIds, List<Long> vectorIds) {
    Map<Long, Double> scores = new HashMap<>();
    addRrfScores(scores, keywordIds);
    addRrfScores(scores, vectorIds);
    return scores.entrySet().stream()
            .sorted(Map.Entry.<Long, Double>comparingByValue()
                    .reversed())
            .map(Map.Entry::getKey)
            .limit(FUSED_RESULT_SIZE)
            .toList();
}
```

代码6-4保留了并行召回与 RRF 融合的主干逻辑。索引初始化、服务可用性检查和异常降级代码未在正文中列出；实际实现中，检索异常会转换为空结果，并由后续数据库检索完成降级处理。

如果向量生成失败，向量检索返回空结果，融合方法继续处理关键词检索结果。如果 Elasticsearch 未返回有效候选，`ChatService` 从 MySQL 查询当前用户及文件夹范围内的知识片段，完成基础匹配后进入同一重排方法。

数据库降级检索的关键判断如代码6-5所示。只有 Elasticsearch 返回了有效片段时才直接进入重排，否则从 MySQL 查询当前用户和文件夹范围内的知识片段。该代码体现了“索引用于加速、数据库作为业务数据源”的实现原则。

代码6-5 检索降级关键代码

```java
List<Long> elasticChunkIds = elasticsearchService.hybridSearch(
        userId, folderIds, question, settings);
List<KnowledgeChunk> chunks =
        loadRankedChunks(
                elasticChunkIds, MAX_RERANK_CANDIDATES);

// Elasticsearch 已返回候选时直接进入统一重排方法
if (!chunks.isEmpty()) {
    return rerankAndDiversify(chunks, question);
}

// 未返回候选时，从 MySQL 读取当前用户范围内的知识片段
List<KnowledgeChunk> all =
        chunkRepository.findExistingByFolderIdInAndOwnerId(
                folderIds, userId);
String rankingQuestion = question;
List<String> terms = searchTerms(rankingQuestion);
List<KnowledgeChunk> ranked = all.stream()
        .filter(chunk ->
                isUsableKnowledgeContent(chunk.getContent()))
        .sorted(Comparator
                .comparingInt((KnowledgeChunk chunk) ->
                        score(chunk.getContent(), terms))
                .reversed()
                .thenComparing(chunk ->
                        chunk.getFile().getUploadedAt(),
                        Comparator.reverseOrder())
                .thenComparing(
                        KnowledgeChunk::getChunkIndex))
        .toList();

int bestScore = ranked.stream()
        .mapToInt(chunk ->
                score(chunk.getContent(), terms))
        .max()
        .orElse(0);
int minimumScore = bestScore <= 0
        ? 0 : Math.max(1, bestScore / 5);

List<KnowledgeChunk> candidates = ranked.stream()
        .filter(chunk ->
                score(chunk.getContent(), terms)
                        >= minimumScore)
        .limit(MAX_RERANK_CANDIDATES)
        .toList();
List<KnowledgeChunk> fallbackCandidates =
        candidates.isEmpty()
                ? ranked.stream()
                        .limit(MAX_RERANK_CANDIDATES)
                        .toList()
                : candidates;
// 两种检索路径共用相同的候选筛选方法
return rerankAndDiversify(
        fallbackCandidates, rankingQuestion);
```

### 6.4.3 答案生成与反馈回写实现

候选筛选完成后，`buildContext()` 按“片段编号、文件名、页码、正文”的格式拼接知识库上下文，每个片段使用 `[1]`、`[2]` 等编号。`buildPrompt()` 将用户问题、知识库上下文和第4章确定的分级知识约束写入模型请求。模型返回内容后，服务将回答文本与本次使用的片段一并封装为响应对象；模型接口未返回有效内容时，系统返回检索提示并保留来源对象。

回答生成后，`ChatService.buildSources()` 将最终使用的知识片段转换为来源对象。来源对象包括引用编号、知识片段编号、文件编号、文件名、页码和上下文摘要，前端通过片段编号定位原始资料内容。

返回答案前，`KnowledgeChunkInteractionService.recordCitations()` 对每个来源执行归属校验，增加 `citeCount`，更新 `lastAccessedAt`，并写入 `CITED` 类型的知识片段事件。用户在来源面板点击“很清楚”时，系统增加 `correctHitCount` 并记录 `FEEDBACK_CLEAR` 事件；点击“忘记了”时，增加 `wrongHitCount` 并记录 `FEEDBACK_FORGOT` 事件。两种反馈都会更新最近练习和最近访问时间。前端收到反馈结果后同步刷新当前对话中的来源统计和知识画像，使单次问答转化为后续诊断可以利用的学习行为。

来源反馈回写代码如代码6-6所示。服务先通过 `requireOwnedChunk()` 校验知识片段归属，再根据反馈类型更新累计字段和事件表，最后返回新的统计值供前端即时刷新。

代码6-6 来源反馈回写关键代码

```java
@Transactional
public ChunkFeedbackResponse recordFeedback(
        Long userId, Long chunkId, ChunkFeedbackType type) {
    // 先校验片段是否属于当前登录用户
    KnowledgeChunk chunk =
            requireOwnedChunk(chunkId, userId);
    Instant now = Instant.now();

    // 更新累计字段，并写入对应的行为事件
    if (type == ChunkFeedbackType.CLEAR) {
        chunk.setCorrectHitCount(
                chunk.getCorrectHitCount() + 1);
        recordEvent(chunk,
                KnowledgeChunkEventType.FEEDBACK_CLEAR,
                true, now);
    } else if (type == ChunkFeedbackType.FORGOT) {
        chunk.setWrongHitCount(
                chunk.getWrongHitCount() + 1);
        recordEvent(chunk,
                KnowledgeChunkEventType.FEEDBACK_FORGOT,
                false, now);
    } else {
        throw new IllegalArgumentException(
                "Unsupported feedback type");
    }
    // 统一更新最近练习时间并返回最新统计
    touchPractice(chunk, now);
    return toFeedbackResponse(chunk);
}
```

```java
private double coverageRate(List<KnowledgeChunk> chunks) {
    // 已反馈或已练习片段数占全部片段数的比例
    return chunks.isEmpty()
            ? 0.0
            : practicedCount(chunks) / (double) chunks.size();
}

private double masteryAverage(List<KnowledgeChunk> chunks) {
    // 未产生反馈的片段不参与平均掌握度计算
    return chunks.stream()
            .filter(chunk -> chunk.getFeedbackCount() > 0)
            .mapToDouble(KnowledgeChunk::getMasteryRate)
            .average()
            .orElse(0.0);
}

private boolean isWeakChunkCandidate(KnowledgeChunk chunk) {
    // 反馈数量不足时暂不作为薄弱点输出
    if (chunk.getFeedbackCount()
            < MIN_DIAGNOSIS_FEEDBACK_COUNT) {
        return false;
    }
    if (chunk.getMasteryRate()
            < WEAK_MASTERY_THRESHOLD) {
        return true;
    }
    return chunk.getMasteryRate()
            < BORDERLINE_WEAK_MASTERY_THRESHOLD
            && chunk.getWrongHitCount()
            > chunk.getCorrectHitCount();
}
```

基础指标计算完成后，服务通过 `reviewPriority()` 和 `riskChunk()` 生成薄弱点的复习排序。具体计算规则已经在第4章给出。本章实现逐个读取知识片段的掌握度、正确与错误次数、引用次数和最近练习时间，计算排序值后封装为响应对象，再按复习优先级降序返回。

趋势查询需要在指定日期计算当时的复习状态，因此 `lastPracticeBefore()` 同时检查知识片段字段和该日期之前的行为事件，避免把查询日期之后的练习记录计入历史结果。

### 6.6.1 错题录入与知识片段关联实现

错题页面使用 `FormData` 提交题干、解析、附件、状态、科目标签和关联片段编号。后端 `MistakeService.create()` 保存 `MistakeQuestion` 及附件记录，附件按用户编号和附件类型分目录存储，并使用 UUID 处理同名文件。PDF 或图片文字可通过 `/api/mistakes/recognize` 提取，临时文件在识别完成后删除。

用户录入或编辑错题时，通过 `/api/knowledge-profile/chunks` 搜索知识片段。后端校验片段归属后写入 `MistakeQuestionChunk`，并使用唯一约束防止重复关联。编辑操作会删除取消选择的关系，只为新选择的片段建立关联。

手动错题首次关联片段时，服务调用 `recordPracticeResult()` 写入一次 `PRACTICE_WRONG` 事件。定制练题加入错题集时，由 `createFromTeacher()` 保留来源片段，并通过 `sourceType` 标记关联来源。

错题关联更新的关键实现如代码6-8所示。服务先删除用户取消选择的关系，再为新片段建立关联，并只把新增关联作为新的错误证据写入学习行为。

代码6-8 错题与知识片段关联关键代码

```java
private void applyLinkedChunks(MistakeQuestion mistake,
                               Long userId,
                               List<Long> chunkIds,
                               MistakeQuestionChunkSourceType sourceType,
                               boolean recordWrongForNewLinks) {
    Set<Long> targetIds =
            new LinkedHashSet<>(normalizeIds(chunkIds));

    // 同步删除和新增的知识片段关联
    removeUnselectedLinks(mistake.getId(), targetIds);
    List<Long> newlyLinked = addNewLinks(
            mistake, userId, targetIds, sourceType);

    // 新增手动关联时记录一次错误行为
    if (recordWrongForNewLinks && !newlyLinked.isEmpty()) {
        chunkInteractionService.recordPracticeResult(
                userId, newlyLinked, false);
    }
}
```

代码6-8保留错题关联更新的主流程。`removeUnselectedLinks()` 负责删除取消选择的关系，`addNewLinks()` 负责检查重复关联、校验知识片段归属并保存新增关系，具体循环和数据访问代码不在正文中展开。

### 6.6.2 错题练习与画像回写实现

错题与知识片段建立关联后，练习结果即可同时写入错题事件和知识片段事件。

错题练习接口为 `GET /api/mistakes/practice`。服务将题目数量限制在1至100之间，并从未完全掌握的错题中随机抽取；用户选择学科时，只在对应科目标签范围内抽取。练习完成后，前端调用 `POST /api/mistakes/{mistakeId}/practice-result` 提交答对或答错结果。

后端首先写入 `MistakePracticeEvent`，保留错题练习的时间序列。随后查询该错题关联的全部知识片段，并调用 `KnowledgeChunkInteractionService.recordPracticeResult()`。答对时增加各片段的正确命中次数并记录 `PRACTICE_CORRECT`；答错时增加错误命中次数并记录 `PRACTICE_WRONG`；两种结果都会更新最近练习时间。

如果错题没有关联知识片段，系统仍保存 `MistakePracticeEvent`，知识片段更新数量返回0；存在关联时，练习结果继续写入对应知识片段。

错题练习结果回写如代码6-9所示。该方法同时保存错题层面的练习事件和知识片段层面的练习结果，使错题练习历史与知识画像统计保持关联。

代码6-9 错题练习结果回写关键代码

```java
@Transactional
public PracticeResultResponse recordPracticeResult(
        Long mistakeId, Long userId, boolean correct) {
    MistakeQuestion mistake =
            requireMistake(mistakeId, userId);

    // 保存错题层面的练习事件
    MistakePracticeEvent event =
            new MistakePracticeEvent();
    event.setOwnerId(userId);
    event.setMistakeId(mistake.getId());
    event.setCorrect(correct);
    practiceEventRepository.save(event);

    // 查询关联片段，并回写知识片段层面的练习结果
    List<Long> chunkIds = mistakeChunkRepository
            .findByMistakeId(mistake.getId()).stream()
            .map(link -> link.getChunk().getId())
            .toList();
    int updated =
            chunkInteractionService.recordPracticeResult(
                    userId, chunkIds, correct);
    return new PracticeResultResponse(
            mistake.getId(), correct,
            updated, linkedChunks(mistake));
}
```

## 6.9 本章小结

本章完成了用户学习档案、资料管理、智能问答、学习画像、错题复盘和学习计划等模块的具体实现。资料经过文本抽取和知识切片后形成 `KnowledgeChunk`；问答引用、来源反馈和错题练习通过 `KnowledgeChunkEvent`、`MistakeQuestionChunk` 和 `MistakePracticeEvent` 回写学习行为；画像模块据此更新掌握度、薄弱点和复习优先级；画像建议及计划草稿经用户确认后写入 `StudyPlanItem`。用户后续产生的新反馈再次进入画像计算，使资料、问答、错题、画像和计划之间形成持续更新的数据闭环。

这一实现使系统不再停留在文件上传、分类和查询等静态资料管理功能上，而是能够根据用户的问答反馈与练习结果更新学习状态，并将分析结果转化为下一阶段的复习任务。系统由此具备了“学习行为记录—知识状态分析—复习计划调整”的动态学习支持能力。

从实现结果看，各模块仍按照第4章确定的前后端分离架构和功能边界运行，知识片段承担资料检索、来源反馈、错题关联和画像计算之间的数据连接作用，MySQL 保存业务数据，Elasticsearch 提供检索支持，模型生成的计划需要用户确认后才能写入。上述实现验证了前文提出的模块化架构、知识片段数据中枢、个人数据隔离和人工确认机制，整体符合系统架构与学习闭环的设计目标。
